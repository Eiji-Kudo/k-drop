## 対応済みの懸念点

<details>
<summary>1. router.load()の実行タイミング（HIGH / 修正済み）</summary>

| 項目 | 内容 |
|------|------|
| 重要度 | **HIGH** |
| ファイル | `app/v2/src/__tests__/App.test.tsx` (行13-14) |
| 検出者 | routing-reviewer, testing-reviewer |

**問題点**:

`render()` を先に呼び、その後で `await router.load()` を呼んでいる。TanStack Router公式ドキュメントでは、`await router.load()` を `render()` の前に呼ぶことが推奨されている。

```tsx
async function renderRoute(path: string) {
	const router = createAppRouter(
		createMemoryHistory({
			initialEntries: [path],
		}),
	);

	render(<RouterProvider router={router} />);  // L13: 先にrenderする
	await router.load();                          // L14: 後からload()する
}
```

この順序が問題になる具体的シナリオ:

1. **act warning の発生**: `render()` が完了した時点ではルーターはまだ初期化途中の状態にある。その後に `await router.load()` が解決すると、Reactコンポーネントツリー内で状態更新が発生するが、この更新は `act()` の外で起きる。結果として `Warning: An update to X inside a test was not wrapped in act(...)` が出力される。現状のテストはシンプルなため顕在化していないが、ルートに `loader`（非同期データ取得）や `beforeLoad` ガードを追加した時点で発現する可能性が高い。

2. **Flaky test の原因**: `router.load()` の解決タイミングとReactのレンダリングサイクルが競合すると、テスト実行環境の負荷状態によって `findByText` が要素を見つけられるタイミングが変動する。CIのように並列実行する環境では間欠的に失敗する原因になりうる。

3. **初期レンダリング時のフラッシュ**: `render()` 時にルーターがまだロード完了していないため、`defaultPendingComponent` やフォールバックUIが一瞬レンダリングされてから実際のルートコンポーネントに切り替わる。テストで意図しないDOM状態をアサートしてしまうリスクがある。

**推奨対応**:

`await router.load()` を `render()` の前に移動する。これはTanStack Router公式のテストパターンに準拠する。

```tsx
async function renderRoute(path: string) {
	const router = createAppRouter(
		createMemoryHistory({
			initialEntries: [path],
		}),
	);

	await router.load();
	render(<RouterProvider router={router} />);
}
```

変更点は行の入れ替えのみ。`render()` の時点でルーターは完全に初期化・ロード済みとなるため、act warningは発生せず、初期レンダリングで正しいルートコンポーネントが即座に表示される。

</details>

## 対応不要の懸念点

<details>
<summary>2. @tanstack/react-router と @tanstack/router-plugin のバージョン不一致（MEDIUM / 対応不要）</summary>

| 項目 | 内容 |
|------|------|
| 重要度 | **MEDIUM** |
| ファイル | `app/v2/package.json` (行21, 28) |
| 検出者 | routing-reviewer, config-quality-reviewer |

**問題点**:

`react-router` がマイナー `168`、`router-plugin` がマイナー `167` で、1バージョン差がある。いずれもキャレット(`^`)なしのピン指定のため、pnpm installで自動的に揃うことはない。

```json
"@tanstack/react-router": "1.168.10",
"@tanstack/router-plugin": "1.167.12",
```

TanStack Routerのエコシステムでは、`react-router` と `router-plugin` は同じリリースサイクルで公開される。起きうる問題:

1. **routeTree.gen.ts の型不整合**: `router-plugin` が生成する `routeTree.gen.ts` は、プラグイン自身が認識するルーター型定義に基づく。`react-router` 側で型シグネチャが `1.168.x` で変更されていた場合、生成コードが古いインターフェースを前提としたものになり、`tsc` でコンパイルエラーが発生する可能性がある。
2. **ランタイムの微妙なバグ**: `router-plugin` がコード生成時に埋め込むAPIの呼び出し方が、新しいバージョンの `react-router` と整合しない場合、ルーティングの挙動が想定外になりうる。

**推奨対応**:

`router-plugin` を `react-router` と同じバージョンに揃える。

```bash
cd app/v2
pnpm add @tanstack/react-router@1.168.10
pnpm add -D @tanstack/router-plugin@1.168.10
```

更新後、`pnpm run dev` で `routeTree.gen.ts` を再生成し、`pnpm run check:all` で問題がないことを確認する。

**対応不要の理由**: `@tanstack/router-plugin` の最新リリースが `1.167.12` であり、`1.168.10` は存在しない。両パッケージは別々のバージョン番号で公開されており、peer dependency (`^1.168.10`) も満たしているため、現状で問題ない。

</details>

<details>
<summary>3. routeTree.gen.tsのgit管理方針が不明確（MEDIUM / 対応不要）</summary>

| 項目 | 内容 |
|------|------|
| 重要度 | **MEDIUM** |
| ファイル | `app/v2/src/routeTree.gen.ts`, `app/v2/.gitignore` |
| 検出者 | routing-reviewer, config-quality-reviewer |

**問題点**:

`routeTree.gen.ts` はTanStack Router pluginが自動生成するファイルで、ファイル冒頭にも以下の記述がある:

```ts
// This file was automatically generated by TanStack Router.
// You should NOT make any changes in this file as it will be overwritten.
```

現在、このファイルはgitにコミットされている。ESLintとBiomeでは除外済み:
- `eslint.config.js`: `globalIgnores(["coverage", "dist", "src/routeTree.gen.ts"])`
- `biome.json`: `"includes": ["**", ... "!src/routeTree.gen.ts"]`

しかし `.gitignore` には記載がなく、git管理方針が不明確。

CIスクリプト（`package.json`）の実行順序が問題:

```json
"ci": "pnpm run test && pnpm run test:types && pnpm run lint && pnpm run format:check && pnpm run build"
```

`test` と `test:types` が `build`（Vite起動でファイル生成）より先に走るため、gitignoreに追加してリポジトリから除外すると、CI環境でテストと型チェックが即座に失敗する。

**推奨対応**:

**選択肢A: コミットする（現状維持 / 推奨）**
- メリット: CIの実行順序を変更する必要がない、`git clone` 直後から `tsc` が通る
- デメリット: PRのdiffにノイズが混じる、マージコンフリクトのリスク

**選択肢B: gitignoreに追加し、CIで事前に生成する**
- メリット: リポジトリがクリーン、マージコンフリクト回避
- デメリット: CIスクリプト修正が必要

現時点ではルート数が少なく開発者も限られるため、**選択肢A（現状維持）** が実用的。ルート数が増加した段階で選択肢Bへ移行を検討する。

**対応不要の理由**: 推奨通り選択肢A（現状維持）を採用。意図的なコミット方針として問題ない。

</details>

<summary>4. Vite 8との互換性確認（MEDIUM / 対応不要）</summary>

| 項目 | 内容 |
|------|------|
| 重要度 | **MEDIUM** |
| ファイル | `app/v2/pnpm-lock.yaml` |
| 検出者 | config-quality-reviewer |

**問題点**:

`@tanstack/router-plugin@1.167.12` のpeerDependenciesにおけるviteの条件:

```yaml
peerDependencies:
  vite: '>=5.0.0 || >=6.0.0 || >=7.0.0'
```

明示的に `>=8.0.0` が含まれていない。semver的には `>=5.0.0` が8.xを包含するためpnpmの依存解決は成功するが、このpeerDependency定義の書き方から、plugin作者がメジャーバージョンごとに動作確認を追加する運用であり、Vite 8はまだ検証対象に含まれていない可能性がある。

具体的なリスク:
1. `@tanstack/router-plugin` は `unplugin@2.3.11` を介してViteのplugin APIにアクセスする。Vite 8でplugin hookのシグネチャが変更された場合、ルートツリー自動生成に影響が出る可能性がある。
2. Vite 8のHMR APIに破壊的変更があった場合、ルートファイル追加・変更時のホットリロードが正常に動作しない恐れがある。

**確信度**: 低い。現時点でビルドが通っているなら実害はない。

**推奨対応**:

CIで `pnpm run ci` がすべてpassしていることをもって、現在のバージョン組み合わせでの互換性は確認済みとする。`@tanstack/router-plugin` のpeerDependenciesに `>=8.0.0` が追加されたバージョンに更新されたらアップデートする。

**対応不要の理由**: 確信度が低く、現状ビルドが正常に動作している。理論上のリスクのみ。

</details>

<details>
<summary>6. 404テストの暗黙的前提（MEDIUM / 対応不要）</summary>

| 項目 | 内容 |
|------|------|
| 重要度 | **MEDIUM** |
| ファイル | `app/v2/src/__tests__/App.test.tsx` (行24-28), `app/v2/src/routes/__root.tsx` |
| 検出者 | testing-reviewer |

**問題点**:

`__root.tsx` でルートレベルの `notFoundComponent` が定義されている:

```tsx
export const Route = createRootRoute({
	component: RootComponent,
	notFoundComponent: NotFoundPage,
});
```

テストは `/missing` にアクセスしたとき `NotFoundPage` が表示されることを検証:

```tsx
it("renders the 404 page for an unknown path", async () => {
	await renderRoute("/missing");
	expect(await screen.findByRole("heading", { name: "Page not found" })).toBeInTheDocument();
	expect(screen.getByText("お探しのページは見つかりませんでした。")).toBeInTheDocument();
});
```

このテストは `createRouter` の `notFoundMode` がデフォルトの `"fuzzy"` であることに暗黙的に依存している。`router.ts` では `notFoundMode` を明示的に設定していない。

- **fuzzy（デフォルト）**: マッチしないパスは最も近い親ルートの `notFoundComponent` で処理
- **root**: 常にルートルートの `notFoundComponent` にフォールバック

現在のフラットなルート構造（`__root__` → `/` のみ）では差異は出ないが、将来ネストルートが追加された場合（例: `/dashboard/nonexistent`）、fuzzyモードでは `/dashboard` レイアウトルートの `notFoundComponent` が優先されるため、テストの前提が崩れる可能性がある。

**推奨対応**:

現時点では実害なし。ルート構造が複雑化する段階で `notFoundMode` を明示的に設定することを推奨:

```ts
export function createAppRouter(history?: RouterHistory) {
	return createRouter({
		routeTree,
		defaultNotFoundComponent: NotFoundPage,
		...(history ? { history } : {}),
	});
}
```

**対応不要の理由**: 現時点でルート構造がシンプル（フラット）で実害なし。将来の検討事項として記録。

</details>

<details>
<summary>5. テスト間のルーターインスタンス分離（MEDIUM / 対応不要）</summary>

| 項目 | 内容 |
|------|------|
| 重要度 | **MEDIUM** |
| ファイル | `app/v2/src/router.ts` (行11), `app/v2/src/__tests__/App.test.tsx` |
| 検出者 | routing-reviewer, testing-reviewer |

**問題点**:

`router.ts` にシングルトンルーターと `declare module` による `Register` 型登録があるが、テストでは毎回 `createAppRouter()` で新しいインスタンスを生成している。`Register` はTypeScriptの型レベルの機構であり、ランタイムには一切影響しない。テストで `createAppRouter()` を毎回呼び出す設計は、テスト間の状態汚染を防ぐ正しいパターン。

**チーム内議論**:

routing-reviewer、testing-reviewerともに低確信度で指摘。リードによる判定: `Register` はtype-levelのみでランタイム影響なし。現在の設計は正しく、対応は不要。

</details>
